Operations

Reloads, logs, caches, and real-world failures.

This page focuses on the questions that appear right after first deployment: how to validate config, what to inspect when things break, and how static routes and protected pages behave under nginx.

Validation

What is the safe reload sequence?

Validate first with nginx -t. If the syntax and includes pass, trigger a reload. That keeps the master process alive while workers pick up the new configuration.

Observability

Which logs matter most during troubleshooting?

The access log shows request patterns and status codes. The error log tells you when nginx could not read a file, reach an upstream, or parse configuration the way you expected.

404s

Why did direct HTML routes return 404 before the fix?

Because nginx serves static files literally. If /faq.html or /operations.html do not exist in the image, there is nothing to return. The proper fix is to ship those files and link them intentionally.

Auth

How are the protected pages enforced?

nginx uses a dedicated location block for /private/ and checks credentials against an embedded .htpasswd file. Requests without valid credentials receive the normal HTTP basic auth challenge.

Performance

What lightweight performance tweaks are enabled?

Text responses are gzip-compressed, static assets under /assets/ receive cache headers, and nginx keeps a dedicated health endpoint separate from the HTML routes.

Next steps

What would you add for a more realistic deployment?

TLS termination, upstream services, explicit readiness probes, structured logging, and externalized auth secrets are the next practical steps once the dummy portal graduates from a static demo.
