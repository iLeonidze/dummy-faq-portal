Getting started

How this dummy service is structured.

Use this page when you want the filesystem layout, public routes, health endpoint, and the reasoning behind the simple nginx setup.

Route design

Which public pages are expected to work directly?

The primary public entry points are /, /faq.html, /getting-started.html, and /operations.html. The deeper reference pages under /pages/ stay available as extended reading.

Health probes

Where should a container or Kubernetes probe point?

The image exposes /healthz, which returns a plain 200 OK response. That avoids tying health checks to a specific HTML page and keeps the probe contract stable.

Protected demo pages

Which pages require basic auth?

The protected demo section is mounted at /private/. The included protected pages are /private/, /private/status.html, and /private/runbook.html. The hardcoded demo credentials are test/test.

Image layout

What does the image copy into the container?

The Docker image copies a custom nginx.conf, the static site under /usr/share/nginx/html/, and a dedicated .htpasswd file used to protect the /private/ routes.

404 prevention

Why were the missing page routes a problem?

nginx serves static files literally. If /faq.html or /operations.html do not exist in the image, there is nothing to return. The fix is to ship the files that the portal promises and link them consistently.
