Configuration FAQ

Editing, routing, and reloads

Files

Where does nginx keep its main configuration?

The main file is commonly named `nginx.conf`. Its exact path depends on how `nginx` was installed, but package-based Linux installs often keep it under `/etc/nginx/`. That main file can include extra files such as per-site or per-service configuration snippets.

Server blocks

What does a server block do?

A `server` block defines how `nginx` should respond for a given hostname, port, or socket. This is the basic unit for hosting multiple sites or APIs from the same instance.

Request matching

How do location rules decide where a request goes?

`nginx` evaluates `location` blocks according to its matching rules: exact matches have priority, prefix matches are considered, and regular expression locations are checked in their own stage. The exact outcome depends on the combination of prefixes and regex rules, so it is worth keeping those definitions simple and intentional.

Validation

How should I validate and reload nginx safely?

A common operational pattern is to run `nginx -t` to validate the syntax and then reload the process after the test passes. Reloading tells the master process to apply the updated configuration without dropping established traffic in the same way a hard restart would.

Observability

Which logs matter first during troubleshooting?

Start with the access log to confirm request patterns and status codes, then inspect the error log for parse problems, upstream failures, permission issues, and module-related messages. Those two logs solve a large share of first-line debugging cases.

Static delivery

When should I use root, alias, or try_files?

Use `root` when a URL path maps onto a directory tree. Use `alias` when the request path needs to map to a different filesystem path. Use `try_files` when you want `nginx` to check multiple candidate files before returning a result or fallback response.
