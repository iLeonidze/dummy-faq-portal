Performance FAQ

Proxying and efficiency

Reverse proxy

Why put nginx in front of an application server?

`nginx` can terminate client connections, serve static content, and forward dynamic requests to upstream services. That lets the app server focus on business logic while `nginx` handles the edge-facing HTTP work.

Compression

Should static assets be compressed?

Usually yes for text-heavy assets such as HTML, CSS, JavaScript, and SVG. `nginx` can gzip responses so browsers transfer fewer bytes. Binary assets like JPEG and PNG generally see little benefit because they are already compressed.

Workers

What does worker_processes auto actually mean?

It lets `nginx` choose a worker count based on the available CPU cores. That is a good baseline for general use because it avoids hard-coding a number that only fits one machine shape.

Caching

How does browser caching help a static portal?

When browsers are allowed to reuse CSS and JavaScript assets, repeat visits feel faster and the server handles fewer redundant downloads. Cache directives need to match your deployment style so users still pick up changed assets when you release an update.

Delivery path

Which core settings show up often in performance discussions?

Settings such as `sendfile`, keepalive timeouts, worker connection limits, and proxy buffering appear often because they shape how efficiently `nginx` moves data between disk, network sockets, and upstream services.

Next steps

What should I learn next for a real deployment?

After the basics, focus on TLS termination, upstream health and failover behavior, cache strategy, and observability. Those topics matter quickly once the server starts fronting real traffic and carrying operational responsibility.
